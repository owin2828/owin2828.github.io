[AWS] VPC 만들어보기(실습)

AWS 기초 - 3

Posted by owin2828 on 2020-01-13 14:31 · 7 mins read

본 문서는 버전 업그레이드를 통해 다시 작성될 예정입니다.

AWS 기초

  • 모든 것이 가상화

    • 하드웨어 관리 비용이 0

    • 하드웨어적 장애가 생겼을 때, 우리가 할 수 있는 것이 없고 AWS가 해결해 주기를 기다려야..

      생각보다 이 문제가 굉장히 스트레스를 받는 문제가 됨

  • 프라이빗 클라우드

    • 하드웨어 위에 가상화 레이어, 그 위에 게스트 서버를 생성해서 사용
    • vmware가 제일 유명, 오픈스택도 존재
    • 하드웨어관리자 — 하이퍼바이저 — 소프트웨어관리자(시스템엔지니어) 이렇게 되는데, 하이퍼바이저를 누가 관리할 것인지가 또 다른 비용의 발생
    • OS 라이센스는 전혀 줄어들지 않음: 각 OS별로 라이센스를 사야함 - > 기존 환경과 차이가 없음
    • 부장님 개인적인 의견으로는 프라이빗 클라우드는 하지 않는 것이,,, -> 일반 사기업이 하기에는 비용절감 효과가 크게 없고, 일만 복잡해짐.
    • 조달된 하드웨어와 가상화된 레이어 위에서 돌기 때문에 시간의 절약은 있지만, 실제 하드웨어를 추가하고 관리하는 비용은 같음

  • 퍼블릭 클라우드

    • AWS의 가장 큰 단위는 region(논리적인 단계), 그 밑에 AZ(Availability Zone, 가용성 범위)

      • AZ가 IDC단위
      • AWS 정책은 하나의 region에 2개 이상의 AZ가 반드시 존재
      • 이것을 싱글AZ로 할건지, 멀티 AZ로 할건지는 사용자가 설정

    • VPC(Virtual Private Cloud)가 AZ 밑의 단위(논리적 구조, IDC 같은 것)

      • 프라이빗이 붙은 이유는 논리적인 하나의 IDC를 만들어서 개인이 사용할 수 있기 때문
      • 우리가 작업하는 모든 리소스는 여기, VPC에 저장
      • 한 region에 5개까지 만들 수 있고, 더 만들고 싶으면 제한용량 해제 시청을 따로 하면 가능

      • VPC 없이도 서버를 생성할 수는 있지만, 권장하지 않음(옛날 방식)

      • 게이트웨이, 라우터, NAT, 서버, 방화벽등으로 구성 (IDC 구성하는 방식으로 구성 가능)

      • 계정을 생성하면 기본 VPC가 생성되고, 그 안에 기본 적인 구성요소들이 생성

        • 이러한 기본 VPC는 건들지도 말고 수정하지도 말자(어떤 오류가 생길 수 있음)

      • 내부적으로 원하는 양만큼 네트워크 설정이 가능

      • VPC 하나에 여러개 서브넷 생성가능(10.1.10.100 10.1.10.110 등..)

        • 10.1.100 이상은 외부로 나가는거, 100 아래는 내부 통신용 처럼 서브넷을 구분하여 설정할 수 있음
        • 서브넷의 설정이 내부적으로 영역을 구분하는 방법
        • 서브넷끼리도 어떻게 연결할지 설정가능

      • 부장님은 VPC별로 개발용, staging용, product용 3개의 VPC를 만드심

        • 아니면 하나의 VPC 안에서 서브넷을 나눠서 그룹별로 분리

          이러한 설정은 나중에 커졌을 때, 서브넷간의 설정이 너무 복잡해져서 VPC를 따로 만드는 것을 추천

        • 아니면 부서별로 VPC를 주는 방법도 존재

        • 어떤 설계든 최대한 VPC를 사용하는 것을 추천

      • VPC(10.1.10) 의 구성

        • ACL(방화벽): 서브넷의 설정을 위한 방화벽
        • 서브넷(10.1.10.100, 10.1.10.101등..)
          • SG(Security Group): 서브넷 하나의 보안을 설정
          • 퍼블릭인지 프라이빗인지를 설정가능
          • 프라이빗 서브넷은 외부에서 들어갈수도, 나갈수도 없음
        • 구성할 때는 전부 다 프라이빗으로 만들고, 하나만 관리용 퍼블릭 서브넷을 구성
          • 퍼블릭으로 한다면, 서브넷을 하나 만들때마다, key를 만들고 설정이 필요한데, 관리자가 n명일 경우 한명만 퇴사해도 모든 서브넷의 키를 바꿔줘여함
            • 따라서 이러한 문제를 방지하기 위해, 퍼블릭을 하나만 만들고 이 퍼블릭으로 ssh로 붙어서 내부적인 프라이빗 서브넷들을 관리
            • 이렇게 하나의 퍼블릭 서브넷을 바스천 서브넷이라하며, ACL과 SG를 설정해서(ACL은 주로 안건드리므로 SG를 조정) 한정된 IP만 이 서브넷에 접근하도록 설정
          • 이렇게 만들어진 서브넷 하나에 만드는 서버를 EC2라고 함(우리가 구성하는 가상 서버, EC2 인스턴스)
            • 오브젝트를 만들고 메모리에 올려서 사용할 수 있는 것을 인스턴스라 하므로, 가상화된 환경에 올려서 사용자가 사용할 수 있도록 하므로, EC2 인스턴스라 함
          • 서브넷 안의 OS등의 업데이트를 위한 퍼블릿 서브넷을 하나 만들고, 이 퍼블릭은 outbound만 가능하게, inbound는 불가능하게 만들고, 외부에서 업데이트를 받아옴
            • 이때 내부 서브넷의 Inbound는 방금 만든 퍼블릭과 붙히고, outbound는 불가능하게, 이러면 OS업데이트를 받아올 수 있음
            • 이렇게 만드는 퍼블릭 서브넷을 NAT라고 함
        • ELB: 로드밸런싱과 보안 설정을 해줌
          • 프라이빗 서브넷을 만들어서 그 안에 웹서버를 두개 만들면, 얘네 들은 외부로 연결이 안됨
          • 이때, ELB를 태워서 외부에서 접근하면 로드밸런싱과 허용된 메서드만 접근가능
          • ELB의 종류는 3개
            • Classic: 옛날 방식
            • Application: 도메인 하위 레벨, 즉 루트 뒤에 /users, /payment 등 다르게 어플리케이션을 웹서버에 구성하고 이에 따라서 밸런싱을 함
            • Nextwork: https, http말고 모든 프로토콜에 대하여 밸런싱을 함
              • 게임은 프로토콜을 각자 커스터마이징해서 사용하므로, Http종류 이외의 프로토콜을 사용하는 경우에 밸런싱하기 편함

    • S3(Simple Storage Service)

      • VPC 밖에 구성된 하나의 웹서버
      • 여기에 .html 같은 정적인 페이지를 두고 앤드 유저가 이곳을 보도록 설정
      • 여기서 브라우저, 클라이언트 단에서 JS로 해당 URL을 호출하도록 설정
      • 이렇게되면, 앤드유저는 VPC를 전혀 신경쓸 필요 없이 그냥 S3의 html만 보면 그것을 보고 쭉 VPC안에 웹어플리케이션 서버까지 접속해서 정보를 들고 올 수 있음
      • 사실상 업계의 표준, 다른 오프젝트들이 이러한 S3와 호환되는 프로토콜을 제공

    • DB

      • 서브넷을 설치하고 그 안에 DB 서버를 구축할 수도 있지만, 그렇게 하지 않고 AWS에서 제공하는 Paas중의 이용
      • 이를 RDS(Relation DB Service)라 하며 이렇게 구성하면 속도도 더 빠름

AWS 실습

  • 콘솔로그인
  • 지역을 서울로 선택
  • 서비스 검색에서 VPC 검색
  • VPC가 default 1개가 존재, VPC 클릭
  • VPC 생성 클릭, 이름 생성 규칙부터 잘 정해야 함
    • object이름/region-code/age-code/ 이런식
    • IOS등을 사용하게되면 IPV6설정을 해주어야 함(나중에 변경이 불가)
  • VPC ID가 나오고, ID를 클릭
  • 생성한 VPC의 정보가 하단에 노출 됨
  • 왼쪽 메뉴에서 ACL을 클릭
    • 방금 생성한 ACL과 기본ACL(서브넷3개짜리)이 존재함을 확인
  • 왼쪽 메뉴에서 인터넷 게이트웨이를 클릭하여 생성된 게이트웨이 확인
    • 이때 Name을 설정해주면서 아까 이름규칙을 적용
    • igw-AP-NE2-TEST 등
  • 왼쪽 메뉴에서 서브넷을 클릭
    • 기본 3개는 두고 새로 생성
    • 이름은 VPC 생성과 같은 규칙
  • ACL메뉴를 클릭하여 다음처럼 서브넷이 연결된것을 확인
  • 왼쪽메뉴에서 보안그룹 생성을 클릭하여 보안그룹을 생성
    • 인바운드 규칙에서 규칙을 추가해서 HTTP를 선택하고, 위치는 내 IP를 설정
    • 인바운드 규칙에서 추가 후, SSH를 선택하고, 위치는 내 IP를 설정
  • 라우팅 테이블 메뉴를 클릭하여 서브넷 연결을 클릭
    • 연결 편집을 클릭하여, 생성한 서브넷을 연결
  • 메인에서 EC2를 검색하여 새로 시작을 클릭
    • 왼쪽메뉴에서 프리티어 클릭해서 보기
    • Amazone 인스턴스2 선택
    • 인스턴스 세부구성에서 네트워크랑 서브넷 설정
    • 스토리지와 태그를 추가한 후, 보안그룹에서 생성한 보안그룹을 선택 후 시작
    • 다음 화면에서는 Keypair를 생성하라는 화면이 나오는데, 이때 생성하는 키페어는 단 한 번만 로컬PC에 저장할 기회를 주므로 주의
      • 만약 잃어버린다면, 새로 생성해야 함
    • 왼쪽 메뉴에서 인스턴스를 클릭 후, 작업에서 연결하기를 눌러 SSH로 접속
    • 서비스 검색에서 S3를 검색 후, 버킷 만들기 클릭
    • 다른 부분은 기본값으로 설정하고, 권한설정에서 퍼블릭 차단을 해제 후 생성
      • 만들어진 s3에는 드래그&드롭으로 파일 업로드 가능
